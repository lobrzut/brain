    CZYM JEST BRAIN

    Brain to portable AI hub w jednym katalogu. Ma trzy ortogonalne role:

    1. Lokalny LLM — Ollama z modelami (qwen2.5, gemma3, embed). Działa offline na Twoim GPU (Vulkan/ROCm dla AMD). Czat przez widget bottom-right albo bezpośrednio przez http://127.0.0.1:11434/v1 (OpenAI-compatible).
    2. Magazyn wiedzy — vault (markdown notatki) + library (PDF/EPUB/MOBI/DOCX/TXT) + semantic search via RAG (embed + sqlite-vec). Wszystko lokalnie, bez chmury.
    3. Most do innych asystentów — przez MCP (serwery brain-vault, brain-library, brain-rag) wpinają się Claude Desktop, Claude Code, Cursor — dają im dostęp do Twojej wiedzy. Plus lokalny proxy do API providerów: jeden klucz w jednym miejscu zamiast w każdym narzędziu.

    Co robi a czego nie

    RobiNie robi
    Czat z lokalnym LLM na Twoim GPU
    Search semantyczny po Twoich książkach i rozmowach
    Destylację historii rozmów do faktów
    Udostępnianie kontekstu LLM-om przez MCP
    Proxy + tracking użycia chmurowych API
    Auto-schedule ciężkich tasków w tle gdy idle    		 Nie syncuje się z chmurą (Twoje dane = Twoje)
    Nie auth-uje się do API web (Claude.ai, ChatGPT.com)
    Nie zastępuje IDE/edytora — vault edytuj w Obsidian
    Nie szyfruje plików (zwykły dysk)
    Nie ma multi-user (single laptop)
    Nie jest SaaS — zero subskrypcji, zero vendor lock-in

    Stos technologiczny

    WarstwaTechnologiaDlaczego ta, a nie inna
    LLM runtime Ollama Jedyne rozwiązanie z pełnym wsparciem AMD (ROCm/Vulkan) bez kompilowania. OpenAI-compatible HTTP API out of box — każdy tool który zna GPT-4, zna też Ollamę bez zmian w kodzie. Automatyczny model management (pull, delete, VRAM TTL). Alternatywa llama.cpp = tylko binary, zero HTTP, brak auto-reload.
    Embed model nomic-embed-text (768 dim) Open weights, działa lokalnie na GPU. Dobra jakość dla technicznego tekstu po polsku i angielsku. 768 dim = kompromis (OpenAI ada-002 ma 1536 ale wysyła dane do chmury i kosztuje). Alternatywa: bge-m3 (lepsza wielojęzyczna, ale 2× wolniejsza).
    Vector DB sqlite-vec Zero infrastruktury. Jeden plik .db — nie ma procesu do uruchamiania, nie ma portu, nie ma Dockera. Chroma, Qdrant, Weaviate to osobne serwery z ich własnymi crashami. Dla osobistej bazy (<5M wektorów) sqlite-vec jest niewykrywalnie wolniejszy. Można backupować jak zwykły plik.
    Destylacja LLM qwen2.5:14b Najlepszy stosunek jakości do VRAM na 14B. Kluczowe: niezawodnie zwraca poprawny JSON nawet przy długich transkryptach — inne modele tej klasy (Mistral, Llama 3.1) ślimaczyły strukturę wyjścia. Działa lokalnie, dane nie wychodzą. qwen2.5:3b = ~3× szybszy ale gubi kontekst przy dłuższych sesjach.
    Dashboard backend FastAPI + uvicorn Async Python z natywnym streamingiem (SSE/ndjson) — kluczowe dla chat widget i live progress bars. Auto-generuje OpenAPI docs. Flask = sync, brak natywnego streamingu. Django = 10× więcej kodu dla prostego API.
    Knowledge graph D3.js force-simulation + Canvas 2D SVG nie daje rady powyżej ~200 węzłów (re-render całego DOM przy każdym ticku fizyki). Canvas 2D = 1660+ węzłów w 60 fps bez problemu — rysuje piksele, nie elementy. Three.js byłby overkill (WebGL, 3D API) dla 2D grafu. D3 force = sprawdzona fizyka z repulsion/attraction, nie trzeba pisać integratora.
    MCP (komunikacja z agentami) Anthropic MCP protocol Standard który wszystkie liczące się narzędzia AI już przyjęły: Claude Desktop, Claude Code, Cursor, Windsurf, Continue.dev. Alternatywa: własne REST API = każdy agent wymaga custom integration. MCP = jedno API, działające wszędzie. Brain wystawia 3 MCP servery przez stdio (brain-vault, brain-library, brain-rag).
    Edytor vault Obsidian (zewnętrzny) Vault to plain markdown na dysku — Obsidian otwiera folder bezpośrednio (File → Open Vault → brain/data/vault). Wikilinki [[nota]] które Obsidian tworzy automatycznie są czytane przez brain do budowania grafu. Alternatywy: Logseq (niekompatybilne linki), Notion (cloud, proprietarny format), Typora (brak grafu/pluginów). Dashboard VAULT tab to tylko podgląd — nie zastępuje edytora, bo markdown potrzebuje prawdziwego UI.
    Własny RAG vs framework ~400 linii Python (rag.py) LangChain i LlamaIndex to frameworki które abstrakcjonizują na tyle, że debugowanie jest trudniejsze niż napisanie od nowa. Brain RAG robi dokładnie to co trzeba: chunk → embed → sqlite-vec → hybrid search (semantic + keyword boost). Zero magic, zero ukrytych retry loops, każdy krok logowalny. Zysk z frameworka: zero. Koszt: 100+ MB dependencies + wersjonowanie piekło.
    Python runtime Portable Python w bin/python/ Kopiujesz brain/ na nowy komputer — Python jest wewnątrz. Nie musisz instalować nic globalnie, nie ma konfliktów z systemowym Pythonem, nie ma venv aktywowania. Cena: ~60 MB extra w folderze. Warto.

    Dlaczego nie Hermes / inne gotowe agenty

    Hermes, AutoGPT, AgentGPT i podobne to agenty zorientowane na wykonywanie zadań — dostają cel, próbują go osiągnąć w pętli tool-use. Brain to co innego: osobista baza wiedzy która wzmacnia agenty których używasz na co dzień. Nie zastępuje Claude Code czy Antigravity — daje im Twoją wiedzę przez MCP. Różnica filozoficzna: Hermes-style = jeden wielki agent który wszystko robi. Brain = warstwa wiedzy + lokalny LLM + integracja z wieloma narzędziami. Możesz używać obu naraz (i powinieneś).

    Stan na teraz

    Dashboard ma 6 zakładek: DASHBOARD (status live), BRAIN (vault browser / knowledge graph / library / dedup), PIPELINE (distylacja / code-index / skills), TOOLS (agents / mcp / logs / backup / scheduler / idle-guard), OPTIONS (klucze API), HELP (ta). Tray icon w pasku Windows. Procesy: Ollama + dashboard (uvicorn) + tray. Knowledge graph: 1660+ węzłów z 1660+ krawędzi, interaktywny canvas z pin mode.

    Filozofia

    Jeden katalog → jeden ZIP backup. Wszystko portable. Modele LLM ważą najwięcej (~10–20 GB), reszta to drobne. Możesz przenieść brain/ na inny komputer, odpalić start.cmd i działa (przy założeniu Windows 11 + AMD lub NV GPU + Node.js dla MCP). Twoje dane nie opuszczają Twojego dysku. Żadna firma nie wie czego szukasz, co pamiętasz, z jakimi problemami się zmagasz.

    🧠 brain-rag — SEMANTIC SEARCH (najważniejszy)

    Hybrid search (embedding + keyword) po 1659+ notatkach z Twoich rozmów AI + PDF-ach. Nie pamiętasz tytułu? Tu pytasz.

    Użyj brain-rag.search_library żeby znaleźć moje notatki o
"konfiguracja WireGuard MikroTik". Pokaż top 3 wyniki z cytatami.
    brain-rag: znajdź wszystko co rozmawiałem o trading pinescript
indicator. Podsumuj decyzje i działające rozwiązania.
    Sprawdź w brain-rag.library_status ile mam notatek, potem
search_library "cv praca" — podsumuj 3 najistotniejsze rozmowy
o szukaniu pracy / CV.

    Co zwraca

    • Score (im wyższy tym lepsze dopasowanie — >0.5 świetne, 0.1-0.3 słabe, <0.1 random)
    • Nazwa pliku + numer strony
    • Fragment treści (~300 znaków)
    • Powyżej 0.7 znaczy: keyword w nazwie pliku + match semantic

    💻 brain-rag.search_code — RAG NAD TWOIM KODEM

    Osobny vector index nad source-code z folderów które wskażesz (TOOLS → CODE INDEX → ADD PATH). Działa jak search_library ale tylko po Twoich repozytoriach. Wyciąga też symbole (funkcje, klasy) regex-em — działa dla Python/JS/TS/Rust/Go/Java/C/Pine/RouterOS/PowerShell.

    brain-rag.search_code query="parser jsonl claude code" — pokaż top 5
    Szukam funkcji która parsowała Pine Script.
brain-rag.search_code query="pine script parser indicator" lang=".pine"
    brain-rag.code_status — pokaż które katalogi są zaindeksowane
i ile plików w każdym języku.

    Setup

    1. PIPELINE → CODE INDEX → wklej ścieżkę katalogu projektu
    2. SCAN ALL — pierwszy scan ~1-2 min na 100 plików (embed przez Ollama)
    3. Pyta przez Claude Desktop / Antigravity przez search_code

    ⚡ brain-rag.run_skill — AGENTIC WORKFLOWS

    Brain ma skille (claude-os style, bez Redisa) — gotowe workflow w brain/skills/*.md. Każdy automatycznie zbiera kontekst (RAG + vault), woła model i zapisuje wynik do vault/digests/. Wywołanie z dashboardu: TOOLS → SKILLS → ▶ RUN. Z Claude Desktop / Antigravity:

    Wywołaj brain-rag.run_skill z parametrami:
name = mikrotik-config

Streszcz wynik w 3 punktach.

    5 startowych skilli

    • inbox-process — status inboxa, czy warto teraz destylować

    • mikrotik-config — playbook RouterOS z komendami które user faktycznie testował

    Dodaj własny skill: wrzuć plik my-skill.md do brain/skills/ z YAML front-matter (name, description, model, context, outputs) — zostanie wykryty automatycznie.

    AntywzorzecDlaczego źle

    RAG / LIBRARY (SEARCH PO KSIĄŻKACH)

    Działa, multi-format. PyMuPDF + ebooklib + python-docx + mobi → chunker → Ollama nomic-embed-text (768-dim) → sqlite-vec.

    Wspierane formaty

    FormatStatusUwagi
    .pdf✓ pełne wsparciePyMuPDF, page-by-page
    .epub✓ pełne wsparcieebooklib, chapter-by-chapter
    .docx✓ pełne wsparciepython-docx
    .txt / .md✓ pełne wsparcieraw text
    .html / .htm✓ pełne wsparcieBeautifulSoup
    .mobi / .azw / .azw3⚠ best-effortKindle DRM zabija to często — bez DRM działa
    .zip / .7z✓ auto-extractrozpakowane do _extracted_<nazwa>/ obok archiwum, potem indeksowane
    .rar✗ nie działawymaga zewnętrznego unrar.exe — wypakuj ręcznie lub przekonwertuj na 7z

    Jak używać

    1. Wrzuć pliki dowolnego z powyższych formatów do data\library\ (drag & drop w Explorerze lub OPEN FOLDER w BRAIN tab)
    2. BRAIN → LIBRARY → REINDEX
    3. Pipeline auto-rozpakowuje archiwa, wykrywa formaty, ekstrahuje tekst, embeduje
    4. Progress live: indexing X/Y · current: nazwa
    5. Po zakończeniu: indexed N chunks across M files (extracted K archives)

    Wyszukiwanie z poziomu Claude Desktop

    Po restarcie Claude Desktop ma 3-ci MCP server brain-rag z dwoma narzędziami:

    • search_library(query, top_k=5) — semantyczne wyszukiwanie, zwraca top-K chunków z cytatem (nazwa PDF + nr strony + score)
    • library_status() — lista zaindeksowanych plików, liczba chunków

    Przykład promptu: "Co mówią moje książki o entropii? Użyj search_library."

    Bezpośrednio przez API (test)

    curl -X POST http://127.0.0.1:7860/api/library/search ^
  -H "content-type: application/json" ^
  -d "{\"query\": \"quantum entanglement\", \"top_k\": 3}"

    Z CLI

    "\bin\python\python.exe" ^
  "\pipeline\rag.py" index               REM zaindeksuj
"\bin\python\python.exe" ^
  "\pipeline\rag.py" search "twoje pytanie"
"\bin\python\python.exe" ^
  "\pipeline\rag.py" status              REM ile chunków
"\bin\python\python.exe" ^
  "\pipeline\rag.py" clear               REM wywal indeks

    Pod spodem

    Embed modelnomic-embed-text (768 dim, ~270 MB, szybki na GPU)
    Chunk size~1800 znaków z overlap 200 (próbuje łamać na zdaniach)
    Storagedata\vectordb\library.db (SQLite + sqlite-vec virtual table)
    ReindexPer-PDF: usuwa stare chunki dla danego pliku, tworzy nowe. Reszta indeksu nietknięta.
    Batch size32 chunków na request do Ollamy

    Limity i co dorobić później

    • OCR PDF-ów ze skanami (PyMuPDF zwraca pusty tekst dla plików image-only — trzeba Tesseract) — nie zrobione. Objaw: REINDEX przeskakuje plik jako "0 chunks".
    • Re-rank wyników (np. cross-encoder albo BM25 na top-K) — nie zrobione. Obecny hybrid search = semantic score + keyword boost w nazwie pliku, co już znacznie pomaga.
    • Auto-reindex przy wrzuceniu nowego pliku (watcher) — nie zrobione. Na razie kliknij REINDEX ręcznie po wrzuceniu nowych PDF-ów.

    AUTO SCHEDULE — TŁA

    Brain ma adaptive scheduler — robi ciężkie zadania (re-destylacja, dedup, inbox collect) gdy NIE używasz kompa. Inspirowane Windows Update / OneDrive sync.

    Jak rozpoznaje że jesteś nieaktywny

    • Idle detection — Windows API GetLastInputInfo(): sekundy od ostatniego ruchu myszy/klawiatury. Niezależne od dashboardu — działa nawet jak masz Brain zwiniętego do tray.
    • CPU threshold — task nie odpali się jak system jest obciążony (np. grasz, kompilujesz coś, leci video).
    • Time window — niektóre taski tylko nocą (22:00-06:00), niektóre dowolnie.

    Startowe 4 zadania

    TaskWindowWarunkiCo robi
    redistill_thinanyidle ≥10 min, CPU<40% Co 15 min: 5 krótkich notatek (<500B) re-destyluje na qwen2.5:14b. ... takich notatek → ~...h cierpliwości rozłożone na wiele "wyjść z domu".
    redistill_thin_nightnoc 22-06CPU<95% Nocą — 20 notatek co 20 min. Jedna noc = ~120 notatek przerobionych. ... / 120 ≈ ... noce do skończenia.
    inbox_collectanyidle ≥5 min, CPU<50% Co 6h: sprawdza inbox, zbiera nowe ZIP-y, destyluje świeże sesje.
    dedupe_scannoc— Raz/dobę o 3:00 — odświeża listę duplikatów do merge.

    UI w TOOLS → AUTO SCHEDULE (BACKGROUND WORKER)

    • System info bar: aktualny CPU%, ile sek jesteś idle, czy jakiś task akurat leci
    • Per task: kafelek z badges (NOC / DZIEŃ / IDLE wymagane / CPU<X), opisem, statusem ("gotowe do uruchomienia" albo lista blokerów), checkbox enabled, przycisk RUN NOW (ignoruje warunki)
    • Historia: collapsible, ostatnie 10 uruchomień z statusem OK/ERR + duration + summary

    Konfiguracja

    Plik data/schedule-config.json — możesz ręcznie edytować interval_sec, require_idle_sec, require_cpu_below, window. Plus dodawać własne taski (musisz dorobić handler w pipeline/scheduler.py → ACTIONS).

    Mój praktyczny plan dla ciebie

    1. Zostaw wszystkie 4 taski enabled (default)
    2. Wieczorem zostaw komputer włączony — scheduler ruszy ~22:00 i przerobi 100+ notatek do rana
    3. W dzień gdy wyjdziesz z domu na >10 min (lunch, spacer, pranie) — scheduler dorzuci 5 notatek
    4. Po 3-4 nocach masz vault soczysty

    IDLE GUARD / VRAM

    Ollama domyślnie trzyma model w VRAM jakiś czas po użyciu (~5-30 min, zależnie od OLLAMA_KEEP_ALIVE). To jest celowo: następny request leci instant zamiast czekać 5-10s na ponowne załadowanie.

    Skutek uboczny: VRAM zajęty mimo że nic nie robisz. Na 16 GB karty model 14B zżera 8-10 GB. Inne aplikacje (gry, video) mogą cierpieć.

    Co masz w dashboardzie

    1. Kafelek OLLAMA na DASHBOARD pokazuje teraz IN VRAM · X GB z listą załadowanych modeli + przycisk UNLOAD
    2. TOOLS → IDLE GUARD — auto-unload toggle: po N minutach idle automatycznie wywala wszystko z VRAM
    3. UNLOAD NOW w sekcji IDLE GUARD — natychmiastowy clean

    Co liczy się jako "aktywność"

    • Hit na /proxy/<provider>/... — chmurowe API przez brain
    • (Chat widget i bezpośrednie wywołania Ollamy są niezależne — Ollama sama zarządza swoim TTL)

    Jak modele są tylko w pamięci Ollamy a nie używane przez brain proxy, to Ollama sama wyrzuci po OLLAMA_KEEP_ALIVE. Możesz to skonfigurować edytując start.ps1: $env:OLLAMA_KEEP_ALIVE = "5m" (domyślnie 30m).

    Co NIE jest aktywnością GPU

    30% util GPU na kafelku GPU gdy nic nie robisz to typowo nie Ollama, tylko Windows desktop compositor + przeglądarka z hardware acceleration + Chrome/Edge z otwartą wieloma kartami. To normalne i nieszkodliwe.
